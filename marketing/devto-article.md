## Article Body

If you've written Go for more than a few months, you've probably written the same boilerplate dozens of times: a fluent builder for a config struct, a `Validate()` method that checks required fields, a `Clone()` for safe copying, a mock for an interface so you can unit test without hitting a real database.

I got tired of it. So I built **gcraft** — a CLI that reads your struct and interface definitions and generates that code for you.

## The problem

Here's a typical struct in a real project:

```go
type User struct {
    ID    int
    Name  string
    Email string
    Tags  []string
}
```

To use this safely across a codebase, you usually end up writing:

- A builder, so callers don't need to remember field order
- A validator, so you don't ship a `User` with an empty email
- A `Clone()`, so you don't accidentally share a slice between two callers
- If `User` is behind an interface (like a repository), a mock for testing

That's 60-80 lines of code that's almost entirely mechanical — and it has to be kept in sync by hand every time you add a field.

## How gcraft works

gcraft parses your source file with Go's standard `go/ast` and `go/parser` packages — no third-party parsing library, no reflection at runtime. It walks the AST looking for the type you specified, extracts its fields (for structs) or method signatures (for interfaces), and feeds that into Go's `text/template` package to render the output. The result is passed through `go/format` so it comes out exactly like `gofmt` would write it.

```bash
gcraft generate -type User -src ./user.go
```

This produces `user_gen.go`:

```go
// Code generated by gcraft. DO NOT EDIT.
package main

type UserBuilder struct {
    v User
}

func NewUserBuilder() *UserBuilder {
    return &UserBuilder{}
}

func (b *UserBuilder) WithName(v string) *UserBuilder {
    b.v.Name = v
    return b
}

// ... WithID, WithEmail, WithTags ...

func (b *UserBuilder) Build() User {
    return b.v
}

func (v User) Validate() error {
    if v.Name == "" {
        return fmt.Errorf("User.Name is required")
    }
    if v.Email == "" {
        return fmt.Errorf("User.Email is required")
    }
    return nil
}

func (v User) Clone() User {
    c := v
    if v.Tags != nil {
        c.Tags = make([]string, len(v.Tags))
        copy(c.Tags, v.Tags)
    }
    return c
}
```

For interfaces, gcraft generates a mock with a `*Func` field per method (so you can override behavior per test) and a `*Called` counter (so you can assert call counts without an assertion library):

```go
type MockUserRepository struct {
    FindByIDFunc   func(id int) (*User, error)
    FindByIDCalled int
}

func (m *MockUserRepository) FindByID(id int) (*User, error) {
    m.FindByIDCalled++
    if m.FindByIDFunc != nil {
        return m.FindByIDFunc(id)
    }
    return nil, nil
}
```

## Design decisions

**No runtime reflection.** Everything is generated ahead of time as plain `.go` files. You can read the generated code, diff it in code review, and it compiles like any other file in your package — no magic at runtime.

**No external dependencies.** gcraft only uses the Go standard library: `go/ast`, `go/parser`, `go/format`, `text/template`. This keeps the binary small and avoids dragging dependencies into projects that use it.

**`//go:generate` compatible.** Drop a directive above your type and run `go generate ./...` as usual:

```go
//go:generate gcraft generate -type User -src ./user.go
type User struct { ... }
```
